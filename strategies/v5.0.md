# V5.0 Strategy Explanation

This file explains the current `V5` workflow in simple language.

Important note:

- `V5` in this repo is the MT5 demo-paper workflow for `EURUSD`
- it is separate from the Alpaca `ETH/USD` research path

## What V5 is doing

Think of V5 as 3 connected parts:

1. `MT5 bridge EA`
2. `Python brain`
3. `MT5 executor`

In simple terms:

- MT5 watches the market and your account
- it sends that information to Python
- Python decides whether there is a trade or not
- Python checks strict risk rules
- if approved, Python sends an order command back to MT5
- MT5 places or manages the trade

So the system is not one single bot doing everything by itself.
It is more like:

- MT5 collects live information
- Python thinks
- MT5 executes

## The workflow step by step

### 1. MT5 sends a snapshot

The MT5 bridge sends Python:

- current bid and ask
- spread
- recent `5m`, `15m`, and `4h` candles
- account balance, equity, margin, and trade status
- open trades

This happens continuously, about every second.

### 2. Python waits for a new 5-minute bar

V5 mainly makes entry decisions once a new `5-minute` candle has closed.

That means it is not trying to trade every tick.
It waits for a new bar, then checks if there is a valid setup.

### 3. The entry model gives an opinion

The entry model does not place orders directly.
It only gives one of these answers:

- `enter_long`
- `enter_short`
- `hold`

It also gives:

- confidence
- short reason
- thesis tags
- optional requested risk size

So the model is an advisor, not the final boss.

### 4. The hard risk engine checks the idea

Even if the model wants a trade, the risk engine can reject it.

This is very important.
The system does not blindly trust the model.

### 5. If approved, the planner builds the trade

The planner calculates:

- entry price
- stop loss
- take profit
- lot size

This part is deterministic.
The model does not choose the stop loss or take profit directly.

### 6. MT5 receives the command and executes

If trading is enabled, Python sends the order command to MT5.
Then MT5 places the actual trade.

If shadow mode is on, it only records what it would have done and does not place the trade.

### 7. Open trades are managed

While the trade is open, a second model can suggest management actions like:

- hold
- move stop to breakeven
- take partial profit
- trail the stop
- close the trade

But again, it can only choose from actions that the hard rules allow.

### 8. Closed trades become lessons

When a trade closes, the system records:

- what happened
- whether it won or lost
- simple lessons to avoid or repeat similar situations later

So V5 has a small feedback loop.
It does not self-modify wildly.
It just remembers what recently worked and what recently failed.

## How the strategy decides to enter

This is the key idea:

V5 does not use one simple fixed formula like:

- "EMA cross means buy"
- "RSI below 30 means buy"

Instead, it gives the model a market summary and asks:

"Should we go long, short, or do nothing?"

The summary includes:

- `5m` market behavior for timing
- `15m` direction for confirmation
- `4h` direction for bigger-picture bias
- spread
- volatility
- account state
- current open exposure
- recent lessons from past trades

So in plain language, the entry logic is:

- look at short-term timing
- check whether medium and higher timeframes support it
- avoid trades when the picture is mixed
- avoid trades when spread is bad
- avoid trades when recent feedback says this kind of setup has been weak
- prefer `hold` when things are unclear

This means the strategy is:

- multi-timeframe
- context-based
- model-guided
- risk-controlled by hard code

## What the model is really looking at

The context packet mainly summarizes:

- price change over recent bars
- EMA 9 vs EMA 20 relationship
- ATR volatility
- trend direction on `5m`, `15m`, and `4h`
- whether spread is tight or wide

So if you simplify it, the model is trying to answer:

- Are the short and bigger timeframes aligned?
- Is the market moving clearly enough?
- Is the spread acceptable?
- Have recent trades warned against this type of setup?

If yes, it may choose a trade.
If not, it should hold.

## The real role of the model

The model decides:

- direction
- confidence
- basic trade idea

The model does not decide:

- final permission
- final risk approval
- exact stop loss placement
- exact take profit placement
- exact lot size formula

That part is handled by deterministic code.

So the design is:

- model for judgment
- code for safety

## Risk: how much V5 risks

The base allowed risk per trade is a range:

- minimum `0.25%` of account equity
- maximum `0.75%` of account equity

If the model does not ask for a specific size, the system uses the middle of that range.

### Example

If equity is `$10,000`:

- minimum risk is about `$25`
- maximum risk is about `$75`
- middle is about `$50`

So a normal trade is roughly risking around that amount, depending on the posture and model request.

## Risk posture: when V5 becomes smaller or a little bigger

V5 adjusts risk based on recent results.

It has 3 posture modes:

- `reduced`
- `neutral`
- `mildly_aggressive`

### Reduced

If recent results are weak, V5 cuts risk.

This happens when:

- at least 2 of the last 3 trades lost, or
- the recent average result is clearly negative

Then risk is multiplied by `0.75`.

Example with `$10,000` equity:

- normal midpoint risk about `$50`
- reduced midpoint risk about `$37.50`

### Neutral

This is the normal state.
Risk multiplier is `1.0`.

### Mildly aggressive

If recent results are strong, V5 can slightly increase risk.

This happens when:

- there are at least 3 wins in the last 5 trades, and
- recent average result is strong enough

Then risk is multiplied by `1.10`.

Example with `$10,000` equity:

- normal midpoint risk about `$50`
- mildly aggressive midpoint risk about `$55`

So V5 does adapt risk, but only a little.
It does not double down wildly.

## Hard risk rules that can block an entry

A trade can be rejected for any of these reasons:

- the decision is just `hold`
- symbol does not match the configured runtime symbol
- account mode is wrong
- account is not trade-enabled
- confidence is below `0.60`
- the decision arrived too late
- spread is above `20 bps`
- another command for that symbol is already pending
- the daily loss kill switch is active
- the bar already used its new-entry limit
- there is already opposite-direction exposure open
- too many same-direction tickets are already open
- a second stacked ticket is not allowed yet
- total basket risk would be too high

## Daily loss kill switch

V5 has a daily-style protection rule:

- if equity drawdown versus balance reaches `2%`
- new entries are blocked

In simple terms:

- if the account is taking too much damage
- stop opening new trades

## Trade frequency and stacking

V5 is also trying not to overtrade.

It limits entries like this:

- only `1` new entry per `5-minute` bar
- at most `2` same-direction tickets per symbol
- no opposite-direction trade while the other side is open

The second ticket is only allowed if the first trade is already protected.

That means:

- no reckless stacking
- no opening a second trade while the first one is still exposed and vulnerable

## How stop loss is placed

Stop loss is not guessed by the model.
It is calculated by code.

The planner does this:

1. calculate `ATR(14)` from recent `5m` bars
2. multiply ATR by `1.25`
3. clamp the stop distance between `6` and `25` pips

So the formula is roughly:

- `stop distance = ATR x 1.25`
- but never less than `6` pips
- and never more than `25` pips

### Example

If ATR is `8` pips:

- stop distance becomes `10` pips

If ATR is too small:

- stop still stays at least `6` pips

If ATR is very large:

- stop is capped at `25` pips

This means V5 tries to adapt to volatility, but not without limits.

## How take profit is placed

Take profit is also deterministic.

The planner sets:

- `take profit distance = stop distance x 1.25`

So the base reward target is about `1.25R`.

### Example

If stop distance is `10` pips:

- take profit distance is `12.5` pips

So V5 starts with a basic reward target that is a little bigger than the stop.

## Where SL and TP go

For a `long` trade:

- entry is near the `ask`
- stop loss goes below entry
- take profit goes above entry

For a `short` trade:

- entry is near the `bid`
- stop loss goes above entry
- take profit goes below entry

So the system handles long and short in a mirrored way.

## How position size is calculated

Once V5 knows:

- how much money it is willing to risk
- how far the stop loss is

it calculates lot size from that.

In plain language:

- bigger stop = smaller position
- smaller stop = bigger position

This is exactly what proper risk sizing should do.

So V5 does not say:

"always trade 1 lot"

Instead it says:

"size the position so that if stop loss gets hit, the dollar loss stays near the planned risk amount"

## Open-trade management logic

Once a trade is open, V5 has a set of management actions.

The manager is allowed to choose only among valid actions.

### Possible management actions

- `hold`
- `close_ticket`
- `move_stop_to_breakeven`
- `take_partial_50`
- `trail_stop_to_rule`

### When breakeven is allowed

Breakeven becomes allowed when:

- the trade is not yet protected
- unrealized result is at least `+0.25R`

That means once the trade has moved enough in your favor, the system can move the stop to entry and remove the original downside risk.

### When partial profit is allowed

Partial take profit becomes allowed when:

- unrealized result is at least `+1R`
- partial was not already taken
- size is large enough

Then it closes `50%` of the position.

This is a classic idea:

- lock some profit
- let the rest keep running

### When trailing stop is allowed

Trailing becomes allowed when:

- the trade is already protected
- unrealized result is at least `+1R`

Then the system trails the stop using about:

- `0.75 x ATR(14)`

So it tries to stay with a winning move while tightening protection.

### Important safety rule

V5 will not widen a stop.
It only moves protection in a safer direction.

That is good risk behavior.

## What the management model does

The management model does not invent random actions.

It gets:

- current ticket info
- allowed actions
- market context
- recent lessons

Then it chooses among allowed options.

So again:

- model for judgment
- code for boundaries

## Feedback and lessons

When trades close, V5 creates simple lessons like:

- avoid repeating this kind of losing setup
- reinforce this kind of winning setup
- avoid entries when spread was already wide
- reinforce taking partial profits before trailing

These lessons are fed back into later entry and management packets.

So the system is trying to learn in a controlled way from:

- recent winners
- recent losers
- repeated mistakes

## The most honest summary of the strategy

If I had to explain V5 in one simple sentence, I would say:

`V5 is a multi-timeframe, model-guided MT5 paper-trading workflow where the model suggests the trade idea, but hard-coded risk rules control whether the trade is allowed and how it is sized and protected.`

## Important current limitation

There is one very important implementation detail to understand.

The current MT5 bridge is not yet sending full live `R` information properly for open trades.

Right now the bridge sends some ticket fields like:

- `risk_amount_usd = 0`
- `unrealized_r = 0`
- `partial_taken = false`
- empty `context_signature`
- empty `thesis_tags`

This means the intended management logic is smarter than the current live MT5 state feed.

So today:

- initial entries, SL, TP, and risk gating are the most reliable parts
- but the richer `R-based` management logic is only partially wired in live MT5 mode

In plain language:

- the idea for breakeven, partial profit, and trailing is there
- but the live bridge still needs improvement for those features to work fully the way they were designed

## Final plain-English summary

V5 works like this:

- MT5 sends live market and account data
- Python summarizes the market
- the model gives a trade opinion
- hard risk rules approve or reject it
- if approved, code calculates size, stop loss, and take profit
- MT5 executes the order
- the system then watches the trade and may protect, reduce, trail, or close it
- after the trade ends, it records lessons for future decisions

So the core philosophy is:

- let the model read the market
- do not let the model control risk by itself
- keep trade execution structured and auditable
